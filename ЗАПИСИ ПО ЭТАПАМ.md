# Этап 1.
1. Оптимальный стек для MVP (один лучший путь)  
Decision Note: Беру единый TypeScript-стек в monorepo — это самый быстрый и реалистичный путь для одного разработчика, с минимальной когнитивной нагрузкой.
- Monorepo: pnpm workspaces
- Backend: Node.js + TypeScript + Express
- ORM / DB schema: Prisma
- DB: PostgreSQL (open-source)
- Background jobs: BullMQ + Redis (очередь, retries, асинхронный pipeline)
- Frontend: React + TypeScript + Vite + React Router + TanStack Query
- Auth: email/password + JWT (MVP-достаточно)
- Admin/Dev panel: отдельные admin-экраны в том же frontend-приложении (/admin/*)
- Локальная инфраструктура: Docker Compose (Postgres, Redis, Mailpit)
AI policy (по твоим ограничениям):
- Бесплатный/OSS-first через адаптеры.
- Платные провайдеры (например OpenAI) — только в контуре transcription/postprocessing и только как replaceable adapters.
- В коде сразу закладывается абстракция:
  - TranscriptionProvider
  - PostProcessingProvider

Этап 4

Сделано: реализовал upload-контур встречи с валидацией длительности, запуском run/step-логов и передачей в processing pipeline.
- Добавил полноценную логику в POST /api/me/meetings/:meetingId/upload в apps/backend/src/routes/meetings.ts.
- Ограничение по длительности теперь строгое: если файл > MAX_MEETING_MINUTES (по умолчанию 120), API возвращает 400 с понятными полями maxMinutes и actualMinutes, файл удаляется, обработка не стартует.
- После успешного upload:
  - обновляется Meeting (sourceFilename, sourceMimeType, sourceDurationSec, sourceTempPath, status=PROCESSING);
  - создается MeetingProcessingRun со статусом queued (в API отдаю status: "pending" + runStatus: "queued");
  - создается начальный ProcessingStepLog с шагом file_received и status=success;
  - ставится задача в BullMQ на транскрибацию.
- Добавил защиту от накопления исходников:
  - при повторном upload старый sourceTempPath удаляется;
  - в pipeline удаление temp-файла остается после транскрибации (source_deleted);
  - также добавил best-effort cleanup на ошибке пайплайна (apps/backend/src/services/processingPipeline.ts), чтобы файлы не зависали навсегда.
Как работает передача upload -> processing pipeline
- API принимает файл, валидирует mime + длительность через ffprobe.
- В транзакции сохраняет путь temp-файла в Meeting, создает MeetingProcessingRun и стартовый ProcessingStepLog.
- Затем enqueue в processingQueue (process-meeting с meetingId + runId).
- Worker (processMeetingRun) переводит run в running, читает meeting.sourceTempPath, выполняет транскрибацию и последующие шаги.
- После транскрибации удаляет source-файл и обнуляет sourceTempPath; при fail тоже делает cleanup best-effort.
