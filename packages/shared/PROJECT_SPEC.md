# ТЗ V1

## SaaS-сервис для асинхронной работы со встречами

------

## 1. Общая идея продукта

1. Продукт представляет собой SaaS-сервис для асинхронной работы с результатами встреч.
2. Сервис должен позволять пользователю:

- загрузить аудио- или видеофайл встречи;
- получить транскрипцию встречи;
- получить саммари встречи;
- получить автоматически выделенные задачи;
- задавать вопросы ИИ по содержанию встречи;
- экспортировать результаты встречи в Telegram и по email;
- вернуться к результатам встречи в любой момент без необходимости хранить исходный медиафайл.

3. Продукт должен быть реализован как **WEB-приложение**.

4. WEB-приложение должно включать:

- пользовательский интерфейс для работы со встречами;
- серверную часть для обработки данных, хранения результатов и интеграции с ИИ;
- внутренний служебный интерфейс администратора/разработчика.

1. Продукт первой версии ориентирован на **личное использование** пользователем, но внутренняя архитектура должна быть подготовлена для последующего перехода к командному режиму.
2. Внутренняя архитектура должна строиться по модели **Personal-first, Team-ready**.

------

## 2. Цели первой версии

1. Первая версия продукта должна решать следующие основные задачи пользователя:

- сохранить содержательное ядро встречи без хранения исходного файла;
- быстро понять, что обсуждалось;
- увидеть, какие задачи были поставлены;
- уточнить детали встречи через вопросы ИИ;
- быстро отправить результаты встречи в Telegram или по email;
- использовать сервис как рабочую память по своим встречам.

1. Первая версия должна подтвердить ценность следующего базового сценария:
   **встреча → транскрипция → саммари → задачи → вопросы ИИ → экспорт результата**.

------

## 3. Границы первой версии

1. В первую версию входят:

- пользовательские аккаунты;
- личное пространство пользователя;
- загрузка встречи;
- транскрипция;
- саммари;
- выделение задач;
- определение/рекомендация спикеров;
- вопросы ИИ по встрече;
- экспорт в Telegram;
- экспорт по email;
- хранение транскрипции и результатов обработки;
- удаление исходного медиафайла после транскрипции;
- подписка;
- внутренний служебный контур администратора/разработчика.

1. В первую версию не входят:

- полноценные групповые пространства;
- совместный доступ нескольких пользователей к одной встрече;
- сложные роли и права;
- корпоративный кабинет администратора компании;
- seat-based billing;
- SSO;
- аудит действий корпоративного уровня;
- сложные интеграции с внешними системами;
- развитая аналитика для бизнеса;
- продвинутый автоподбор моделей;
- автоматическое A/B-сравнение моделей.

------

## 4. Целевая архитектурная модель

1. Система должна быть реализована как многопользовательская.
2. На первой версии пользователь работает в своем личном пространстве.
3. Архитектурно данные должны принадлежать не только пользователю напрямую, а рабочему пространству (`workspace`).
4. На первой версии для каждого пользователя должен автоматически создаваться **personal workspace**.
5. Каждая встреча должна принадлежать конкретному workspace.
6. Такая архитектура должна обеспечивать возможность дальнейшего расширения до командного режима без существенной переделки базы данных и бизнес-логики.

------

## 5. Основной пользовательский сценарий

1. Пользователь загружает аудио- или видеофайл встречи.
2. Система принимает файл и запускает транскрибацию.
3. После успешного получения транскрипции система:

- сохраняет транскрипцию;
- удаляет исходный аудио-/видеофайл;
- запускает постобработку встречи.

1. Постобработка встречи должна включать:

- суммаризацию;
- выделение задач;
- рекомендации по определению спикеров;
- подготовку данных для вопросов ИИ по встрече.

1. После завершения обработки пользователь в карточке встречи должен получить:

- транскрипцию;
- саммари;
- список задач;
- список спикеров;
- возможность задать вопрос ИИ;
- возможность экспортировать результаты.

1. Пользователь должен иметь возможность в любое время вернуться к обработанной встрече и продолжить работу с ее результатами.

------

## 6. Функциональные требования: аккаунт и доступ

1. Система должна поддерживать регистрацию пользователя.
2. Система должна поддерживать аутентификацию пользователя.
3. У каждого пользователя должен быть личный аккаунт.
4. У каждого пользователя должен быть личный workspace, создаваемый автоматически.
5. Пользователь первой версии должен видеть только свои встречи.
6. В первой версии все встречи должны быть приватными по умолчанию.

------

## 7. Функциональные требования: сущность встречи

1. В системе должна существовать сущность встречи.
2. Встреча должна содержать минимум:

- идентификатор;
- принадлежность к workspace;
- принадлежность к пользователю-инициатору;
- название;
- тип источника;
- статус обработки;
- дату создания.

1. Пользователь должен иметь возможность:

- создать встречу;
- загрузить файл встречи;
- просмотреть список встреч;
- открыть карточку встречи;
- видеть статус обработки встречи.

------

## 8. Требования к исходным файлам

1. Система должна принимать для обработки аудио- и/или видеофайлы встречи продолжительностью **до 120 минут**.
2. Ограничение в 120 минут является обязательным функциональным требованием первой версии.
3. Если файл превышает допустимую длительность, система должна:

- отклонить файл до запуска обработки;
- показать пользователю понятное сообщение о превышении лимита;
- не запускать транскрибацию и постобработку.

4. Проверка длительности файла должна выполняться до начала основной обработки.
5. Система не должна использовать исходные аудио- и видеофайлы как постоянное хранилище данных встречи.
6. Исходный файл должен использоваться только для получения транскрипции.
7. После успешного получения транскрипции исходный файл должен быть удален из постоянного хранения системы.
8. Под успешным получением транскрипции понимается состояние, при котором транскрипция:

- сохранена в системе;
- доступна для дальнейшей постобработки;
- имеет статус успешного завершения шага транскрипции.

1. В системе должны удаляться не только основной загруженный файл, но и временные копии, если они создавались в процессе обработки.
2. Система может хранить технический факт обработки файла в логах, но не должна хранить сам исходный файл после успешной транскрипции.
3. Если транскрипция завершилась ошибкой, поведение по хранению файла должно определяться технической реализацией, но в ТЗ V1 базовым правилом остается: после успешной транскрипции исходный файл не хранится.

------

## 9. Требования к транскрипции

1. Транскрипция должна сохраняться в системе.
2. Пользователь должен иметь возможность просматривать транскрипцию в интерфейсе встречи.
3. Пользователь должен иметь возможность выгружать транскрипцию локально в текстовый файл.
4. Транскрипция должна содержать:

- текст;
- тайм-стемпы;
- разбивку по спикерам.

1. Рекомендуемый формат представления транскрипции:
   `[время] Спикер: текст`.
2. Транскрипция должна храниться как структурированные данные, а не только как единый текстовый блок.
3. Рекомендуется хранить транскрипцию сегментами с привязкой к:

- спикеру;
- началу сегмента;
- окончанию сегмента;
- тексту сегмента;
- порядку следования.

------

## 10. Требования к спикерам

1. Система должна автоматически выделять спикеров в ходе обработки транскрипции.
2. После обработки система должна показывать пользователю список выделенных спикеров.
3. Для каждого выделенного спикера система должна отображать:

- автоматический ярлык, например `Спикер 1`, `Спикер 2`;
- рекомендацию ИИ по возможному имени или роли спикера, если такая рекомендация сформирована.

1. Пользователь должен иметь возможность вручную задать имя спикера.
2. Пользователь должен иметь возможность изменить ранее заданное имя спикера.
3. После ручного подтверждения или изменения имя спикера должно использоваться во всех связанных представлениях встречи:

- в транскрипции;
- в саммари, если применимо;
- в задачах, если спикер фигурирует как ответственный;
- в вопросах ИИ по встрече;
- в экспортируемых материалах, если это предусмотрено логикой экспорта.

1. Система должна хранить как минимум:

- автоматическую метку спикера;
- рекомендацию ИИ;
- подтвержденное пользователем имя.

------

## 11. Требования к саммари

1. Для каждой успешно обработанной встречи система должна формировать саммари.
2. Саммари должно строиться на основе сохраненной транскрипции встречи.
3. Саммари должно быть доступно пользователю в карточке встречи.
4. Саммари должно быть пригодно для чтения человеком и использования в дальнейшей работе.
5. Саммари может быть использовано как часть данных для экспорта в Telegram и по email.

------

## 12. Требования к задачам

1. После анализа встречи система должна автоматически формировать список задач.
2. Каждая задача должна содержать минимум:

- текст задачи;
- ответственного;
- срок исполнения.

1. Если ответственный не определен автоматически, система должна отображать значение:
   **«ответственный не определен»**.
2. Если срок исполнения не определен автоматически, система должна отображать значение:
   **«срок не установлен»**.
3. Список задач должен быть доступен в карточке встречи.
4. Пользователь должен иметь возможность редактировать каждую задачу.
5. Пользователь должен иметь возможность изменить:

- текст задачи;
- ответственного;
- срок исполнения.

1. Система должна воспринимать автоматически выделенные задачи как черновой результат ИИ, подлежащий пользовательскому подтверждению и редактированию.
2. Рекомендуется сохранять как минимум:

- автоматически предложенное значение;
- финальное пользовательское значение.

------

## 13. Требования к вопросам ИИ по встрече

1. Пользователь должен иметь возможность задавать вопросы ИИ в контексте конкретной встречи.
2. Ответы на вопросы должны строиться на основе данных этой встречи.
3. Вопросы пользователя могут касаться, например:

- решений, принятых на встрече;
- обещаний участников;
- рисков;
- задач;
- договоренностей;
- краткого пересказа для руководителя;
- уточнения, кто и что говорил.

1. Система должна сохранять историю вопросов и ответов по встрече, как минимум в базовом виде.
2. Для вопросов по встрече должна использоваться **та же ИИ-модель**, что и для:

- суммаризации;
- выделения задач;
- определения/рекомендации спикеров.

1. В первой версии не допускается использование отдельной активной модели только для вопросов по встрече.

------

## 14. Требования к экспорту

1. Пользователь должен иметь возможность экспортировать результаты встречи в Telegram.
2. Пользователь должен иметь возможность экспортировать результаты встречи по email.
3. Экспорт должен опираться на уже обработанные данные встречи.
4. В составе экспортируемых данных могут использоваться:

- саммари;
- задачи;
- иные структурированные результаты встречи по логике интерфейса.

1. Экспорт должен быть доступен из карточки встречи.
2. История фактов экспорта должна фиксироваться в системе.
3. Для каждого события экспорта должны сохраняться минимум:

- встреча;
- канал экспорта;
- пользователь, инициировавший экспорт;
- дата и время;
- статус выполнения.

------

## 15. Пользовательские статусы обработки

1. Система должна отображать пользователю понятные статусы обработки встречи.
2. Пользовательские статусы должны быть простыми и не перегруженными техническими деталями.
3. Рекомендуемый набор пользовательских статусов:

- файл получен;
- идет транскрипция;
- транскрипция готова;
- идет обработка результатов;
- встреча обработана;
- произошла ошибка обработки.

------

## 16. Подписка и монетизация

1. Первая версия продукта должна поддерживать подписочную модель.
2. Подписка должна быть ориентирована на личное использование пользователя.
3. Должна поддерживаться подписка:

- на месяц;
- и/или на год.

1. В системе должна существовать сущность подписки, связанная с пользователем и/или его personal workspace.
2. Конкретная тарифная сетка может уточняться отдельно, но архитектура должна предусматривать возможность ограничения доступа по тарифу.

------

## 17. Внутренний служебный контур администратора/разработчика

1. Система должна содержать внутренний служебный контур для разработчика/администратора.
2. Служебный контур должен быть доступен только авторизованному администратору/разработчику.
3. Служебный контур должен решать следующие задачи:

- контроль прохождения обработки встреч;
- просмотр статусов обработки;
- просмотр ошибок;
- просмотр стоимости обработки;
- управление активными ИИ-моделями;
- ручной перезапуск обработки.

------

## 18. Логика использования ИИ-моделей

1. В системе должно использоваться два логических контура выбора моделей:

### Контур 1. Транскрипция

1. Для транскрипции должна использоваться конфигурация типа `transcription`.
2. Для транскрипции может использоваться:

- либо та же модель/провайдер, что и в контуре постобработки;
- либо отдельная модель/провайдер.

### Контур 2. Постобработка встречи

1. Для следующих операций должна использоваться **одна общая активная модель**:

- суммаризация;
- выделение задач;
- определение/рекомендация спикеров;
- ответы на вопросы по встрече.

1. Для указанных операций в первой версии не допускается независимая настройка отдельных активных моделей.
2. В системе должен использоваться единый тип конфигурации `postprocessing` для этих операций.

------

## 19. Требования к конфигурации ИИ-моделей

1. Конфигурация ИИ-моделей должна храниться вне программного кода.
2. Смена активной модели не должна требовать изменения исходного кода приложения.
3. Активные модели должны задаваться через конфигурационный слой, доступный из внутреннего интерфейса администратора/разработчика.
4. В системе должны существовать как минимум два логических типа конфигурации:

- `transcription`;
- `postprocessing`.

1. Для каждого типа конфигурации активной может быть только одна модель.
2. Система должна поддерживать хранение нескольких конфигураций одного типа, включая неактивные альтернативы.
3. Минимальные параметры конфигурации модели:

- логический тип;
- провайдер;
- имя модели;
- признак активности;
- дата и время изменения;
- дополнительные параметры при необходимости.

------

## 20. Раздел “Настройки ИИ”

1. Во внутреннем служебном интерфейсе должен быть раздел **«Настройки ИИ»**.
2. Раздел должен содержать минимум два блока:

### Блок 1. Транскрипция

1. Блок должен позволять:

- просмотреть текущий провайдер транскрипции;
- просмотреть текущую активную модель транскрипции;
- сменить активную модель транскрипции;
- сохранить изменения.

### Блок 2. Постобработка встречи

1. Блок должен позволять:

- просмотреть текущий провайдер модели постобработки;
- просмотреть текущую активную модель постобработки;
- сменить активную модель постобработки;
- сохранить изменения.

1. В интерфейсе должно быть явно указано, что модель постобработки используется одновременно для:

- суммаризации;
- выделения задач;
- рекомендаций по спикерам;
- вопросов по встрече.

1. Смена модели в блоке постобработки должна автоматически применяться ко всем этим операциям.

------

## 21. Логирование обработки встречи

1. Система должна вести журнал прохождения обработки встречи.
2. Для каждой попытки обработки встречи должна создаваться отдельная запись запуска обработки.
3. Повторный запуск обработки одной и той же встречи должен создавать новую запись, а не перезаписывать предыдущую.
4. Логирование должно позволять определить:

- была ли успешно выполнена транскрипция;
- была ли успешно выполнена постобработка;
- на каком шаге произошла ошибка;
- какие провайдер и модель использовались;
- сколько заняла обработка;
- сколько стоила обработка;
- для какого пользователя выполнялась операция.

1. В логах администратора обязательно должно быть видно:

- идентификатор пользователя;
- и/или email пользователя;
- и/или иное однозначное обозначение пользователя,
  для которого производилась операция обработки.

------

## 22. Сущность запуска обработки

1. В системе должна существовать сущность, отражающая одну попытку полной обработки встречи.
2. Рекомендуемое имя сущности: `MeetingProcessingRun`.
3. Сущность `MeetingProcessingRun` должна содержать минимум:

- идентификатор запуска;
- идентификатор встречи;
- идентификатор пользователя, для которого выполнялась обработка;
- идентификатор workspace;
- общий статус обработки;
- провайдер транскрипции;
- модель транскрипции;
- провайдер постобработки;
- модель постобработки;
- дата и время начала;
- дата и время окончания;
- общая стоимость;
- краткое описание ошибки при наличии.

1. Общий статус обработки должен поддерживать как минимум значения:

- `pending`;
- `running`;
- `completed`;
- `failed`;
- `partial_failed`.

------

## 23. Логирование шагов обработки

1. Для каждого запуска обработки должен вестись детальный журнал шагов.
2. Рекомендуемое имя сущности: `ProcessingStepLog`.
3. Для каждого шага должны сохраняться минимум:

- идентификатор записи шага;
- идентификатор запуска обработки;
- идентификатор пользователя, для которого выполнялся шаг;
- имя шага;
- статус шага;
- провайдер;
- модель;
- дата и время начала;
- дата и время завершения;
- длительность;
- стоимость;
- код ошибки при наличии;
- текст ошибки при наличии.

1. В первой версии рекомендуется логировать как минимум следующие шаги:

- получение файла;
- запуск транскрипции;
- успешное завершение транскрипции;
- удаление исходного файла;
- определение/рекомендация спикеров;
- генерация саммари;
- извлечение задач;
- подготовка данных для вопросов ИИ по встрече, если выделяется отдельным этапом;
- сохранение результатов;
- завершение полной обработки.

1. Если один из шагов завершился ошибкой, это должно быть отражено отдельным статусом ошибки в журнале шагов.

------

## 24. Логирование стоимости

1. Система должна вести учет стоимости операций, связанных с ИИ-обработкой.
2. Разработчик/администратор должен иметь возможность видеть:

- стоимость транскрипции;
- стоимость постобработки;
- суммарную стоимость обработки встречи.

1. Стоимость должна фиксироваться как минимум:

- на уровне шага;
- на уровне полного запуска обработки.

1. Если провайдер возвращает данные о токенах, минутах аудио или иных единицах биллинга, система должна сохранять эти данные при наличии.
2. Если точная стоимость не была возвращена автоматически, система должна позволять сохранить оценочную стоимость.

------

## 25. Раздел “Журнал обработок”

1. Во внутреннем интерфейсе должен быть раздел **«Журнал обработок»**.
2. Раздел должен содержать список запусков обработки.
3. Для каждой записи в журнале обработок должны отображаться минимум:

- дата и время запуска;
- встреча;
- пользователь;
- общий статус;
- модель транскрипции;
- модель постобработки;
- общая стоимость;
- длительность;
- наличие ошибки.

1. В журнале должны быть базовые фильтры:

- по статусу;
- по пользователю;
- по дате;
- только записи с ошибками.

1. Из журнала должен быть доступен переход в карточку конкретного запуска обработки.

------

## 26. Карточка запуска обработки

1. Во внутреннем интерфейсе должна существовать карточка конкретного запуска обработки.
2. Карточка должна содержать:

- данные о встрече;
- данные о пользователе, для которого выполнялась обработка;
- общий статус;
- провайдер и модель транскрипции;
- провайдер и модель постобработки;
- время начала и окончания;
- общую стоимость;
- список шагов обработки;
- стоимость каждого шага;
- текст ошибки по шагу;
- общий текст ошибки, если есть.

1. Из карточки должна быть доступна команда ручного перезапуска обработки.

------

## 27. Перезапуск обработки

1. Система должна поддерживать ручной перезапуск обработки встречи из служебного интерфейса.
2. При перезапуске должна создаваться новая сущность `MeetingProcessingRun`.
3. История предыдущих запусков должна сохраняться.
4. Перезапуск не должен удалять исторические логи предыдущих запусков.

------

## 28. Разделение пользовательских и технических статусов

1. В системе должны существовать два уровня статусов:

- пользовательские;
- технические.

1. Пользовательские статусы отображаются конечному пользователю в интерфейсе встречи.
2. Технические статусы отображаются только в служебном интерфейсе администратора/разработчика.
3. Технические статусы должны содержать более детальную информацию:

- название шага;
- код ошибки;
- текст ошибки;
- длительность;
- стоимость;
- использованные провайдер и модель;
- пользователь, для которого выполнялась операция.

------

## 29. Фиксация фактически использованных моделей

1. Система должна фиксировать в логах не только активную конфигурацию на текущий момент, но и фактически использованные:

- провайдер транскрипции;
- модель транскрипции;
- провайдер постобработки;
- модель постобработки.

1. Эти данные должны сохраняться в момент запуска обработки.
2. Последующая смена активной конфигурации не должна менять историю уже выполненных обработок.

------

## 30. Ошибки и диагностика

1. При возникновении ошибки система должна сохранять:

- пользователя, для которого выполнялась операция;
- шаг, на котором произошла ошибка;
- дату и время ошибки;
- код ошибки при наличии;
- текст ошибки;
- используемую модель;
- используемого провайдера.

1. Ошибка на одном шаге должна отражаться в общем статусе запуска обработки.
2. Если часть шагов была выполнена успешно, а часть завершилась ошибкой, общий статус может принимать значение `partial_failed`.
3. Служебный интерфейс должен позволять разработчику/администратору определить, на каком уровне произошла ошибка:

- транскрипция;
- определение спикеров;
- суммаризация;
- извлечение задач;
- подготовка данных для вопросов ИИ;
- сохранение результатов.

## 33. Продуктовые принципы V1

1. В основе V1 должны быть зафиксированы следующие принципы:

### Принцип 1

1. Система хранит не исходный медиафайл, а результат обработки встречи.

### Принцип 2

1. Транскрипция является структурированными данными, а не просто текстом.

### Принцип 3

1. Спикеры должны быть управляемыми пользователем.

### Принцип 4

1. Задачи, выделенные ИИ, обязательно должны быть редактируемыми.

### Принцип 5

1. Неопределенность не скрывается, а явно отображается пользователю:

- «ответственный не определен»;
- «срок не установлен».

### Принцип 6

1. Для всех смысловых операций постобработки встречи используется единая ИИ-модель.

### Принцип 7

1. Администратор/разработчик должен видеть управляемый и прозрачный процесс обработки по каждому пользователю.

### Принцип 8

1. V1 должна быть спроектирована с приоритетом использования бесплатного и open-source стека; платные компоненты допустимы только для ИИ-обработки и только при отсутствии приемлемой бесплатной альтернативы.

------

## 34. Ограничения первой версии

1. В первой версии не требуется:

- сложная аналитическая панель;
- графики расходов;
- автоматический выбор оптимальной модели;
- сложный fallback между несколькими провайдерами;
- сложное версионирование промптов;
- развитый корпоративный контур управления.

1. Для первой версии достаточно обеспечить:

- работающий пользовательский сценарий;
- базовый контроль статусов;
- редактирование задач;
- управление именами спикеров;
- экспорт;
- внутренние логи;
- ручную смену моделей;
- ручной перезапуск обработки.

------

## 35. «Технические и архитектурные ограничения»

1. Продукт должен быть построен **максимально на бесплатных инструментах**.
2. Под бесплатными инструментами в рамках ТЗ понимаются:

- open-source решения;
- бесплатные библиотеки;
- бесплатные фреймворки;
- бесплатные базы данных;
- бесплатные инструменты развертывания и локальной разработки;
- бесплатные UI/Backend-инструменты, не требующие обязательной платной лицензии.

1. Использование платных инструментов допускается только в исключительных случаях, когда:

- отсутствует реалистичная бесплатная альтернатива;
- платный инструмент необходим для ключевой функции продукта;
- его использование явно обосновано в технической документации проекта.

1. Для первой версии таким допустимым исключением могут являться только ИИ-инструменты для:

- транскрибации;
- постобработки встречи, включая саммари, задачи, спикеров и вопросы по встрече,

но только в случае, если бесплатная реализация не обеспечивает приемлемого качества, стабильности или трудозатрат.

1. При проектировании и реализации системы разработка должна исходить из принципа:
    **сначала бесплатный/open-source вариант, платный вариант — только как исключение с явной фиксацией причины.**
2. Все внешние платные зависимости, если они будут использованы, должны быть изолированы через конфигурацию и абстракции, чтобы их можно было заменить в дальнейшем.

## 36. Итоговая краткая формулировка V1

1. Первая версия продукта должна представлять собой личный SaaS-сервис для асинхронной работы со встречами, который:

- продукт является WEB-приложением;
- поддерживает загрузку файлов длительностью до 120 минут;
- строится преимущественно на бесплатном/open-source стеке;
- платные зависимости допускаются как исключение в контуре ИИ-обработки;
- принимает аудио- и видеофайлы;
- получает транскрипцию;
- удаляет исходные медиафайлы после успешной транскрипции;
- хранит структурированную транскрипцию с тайм-стемпами и разбивкой по спикерам;
- рекомендует и позволяет редактировать имена спикеров;
- формирует саммари;
- выделяет редактируемые задачи;
- позволяет задавать вопросы ИИ по встрече;
- экспортирует результаты в Telegram и по email;
- использует отдельный контур модели для транскрипции и единый контур модели постобработки для саммари, задач, спикеров и вопросов по встрече;
- предоставляет разработчику/администратору внутренний журнал операций с ошибками, стоимостью, моделями и указанием пользователя, для которого выполнялась обработка.
